# 첫 번째 회고

## 성윤의 회고

### 요약

cli, git : 명령어에 대해 자세하게 들여다보고 반복학습이 필요하다고 느낌
그 외 : 기술을 선택하는 이유를 스스로 묻고 이해하며, 단순히 “되면 넘어가는” 태도에서 벗어나 적극적으로 질문하고 배우는
개발자로 성장해야겠다는 다짐이다.

### cli

리눅스 기반 명령어를 사용하고 배우면서 잊고있었던 기억이 났다. 대학시절 명령어를 단순 이동하고 디렉토리만 만들었다. 하지만 이번 기회로 cli를 통해 디렉토리 명을 변경하거나 파일 생성및 작성을 함으로써 다양한 명령어에 대한 의미와 사용법에 대해 익히고 아직 익숙하지 못해 꾸준히 노력해야겠다는 생각을 했다.

### git

git 또한 개별적으로 공부를 하면서 접한적이 있다. 그 당시엔 단순 코드 커밋을 저장하고 올리는 수준이었다. 대략적으로 어떤기능이 있는지 들어보기만 하고 깊이 파고들지않았던 점에대해 반성해야겠다는 생각이 들었다. 또한 아직까지 커밋이라던지 협업을 할 때 머지를 통해 코드를 합쳐야 하는것은 알지만 브랜치를 어떻게 나누어야 될지 어떻게 해야 더 효율적으로 해야될 지 고민을 해야할 필요성을 느낀다.

### 그 외

선생님께서 "개발자는 왜? 라는 의문점을 품고 살아야한다" 라고 하셨다. 나는 이전에 공부를 하면서 말씀하신 말에대해 영상을 통해 접한적은 있지만 그렇게 하지 않았다. 예를들어 git과 mercury 둘다 분산 버전관리 시스템인데 윈도우에 더 잘맞는 머큐리보다 깃을 많이 사용할까? 라는 의문점 처럼 내가 공부를 하면서 왜 이 기술을 공부해야하고 이 기술을 선택했는지 이유가 없었다. 이러한 부분에서 시켜서 그냥 하는 개발자가 아닌 왜라는 의문과 기술 사용 시에 여러가지 방식이 있다면 그중 왜 그 기술을 선택해야했는지 이유를 찾아보는 사람이 되도록 노력해야겠다는 생각이 들었습니다.

수업시간에 단순히 코드 작성시에 되면 그냥 넘어가버리는 습관이 있는것 같다. 오늘 gitignore을 배울 때 node_module폴더를 추가했을 때 왜 브랜치 수가 줄어드는가 질문을 했던 분이 계셨다. 질문을 듣고 이유는 이번에 배운것을 기반으로 대략적으로 알고있었지만 나보다 더 넓은 시각을 가지고 있구나 라고 생각했고 다른 사람이 에러가 났거나 질문을 할 때 해결방안을 찾아보고 코드를 공유하는 모습을 보고 이에대해 본받아야겠다는 생각을 했다.
